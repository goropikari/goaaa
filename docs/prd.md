# goaaa: Go テスト向け AAA Pattern Linter

## 1. タイトル

Go のテストコードが Arrange–Act–Assert（AAA）パターンに沿っているかを検査する linter

## 2. 要約

Go 開発者がテストの構造を一貫して保てるように、テストコードを静的解析し、AAA の順序や不足を検出する単独実行可能な CLI linter を提供する。

初版では標準 Go パッケージのみを使用する。テーブル駆動テスト、サブテスト、モックやアサーション用ライブラリを使うテストも対象とするが、判定はライブラリ非依存とし、曖昧なケースでは誤検出を抑えることを優先する。

## 3. 課題

テストが準備・実行・検証の責務を混在させると、可読性や保守性が低下する。しかし、コードレビューだけでは AAA の崩れを一貫して検出できない。また、Go のテストにはテーブル駆動や `t.Run`、モック利用など複数の記述形式があり、単純な命令数や関数呼び出しだけでは判定しにくい。

## 4. 目的

- テスト関数の AAA 構造上の明らかな違反を自動検出する。
- テストの可読性・保守性に関する規約を、開発者が CLI で手軽に確認できるようにする。
- 判定根拠が分かる、修正可能な診断メッセージを提供する。

## 5. 非対象

- テストの機能的な正しさ、網羅率、実行結果の検証
- コーディングスタイル全般や命名規則の検査
- 外部サービスへの接続、テストの実行、コードの自動修正
- 初版での `golangci-lint` など既存 linter 基盤への直接統合
- AAA への分類が型情報や実行時情報を必要とする曖昧な処理の断定的な判定

## 6. 対象ユーザー

- Go で開発・テストを行う開発者
- CLI を使ってローカルまたは簡易な開発チェックでテスト品質を確認したいチーム

## 7. スコープ

### 対象

- `TestXxx` と `t.Run` のサブテスト本体
- `// Arrange`、`// Act`、`// Assert` マーカーによるフェーズ区切り
- `t.Run` を使ったサブテスト
- テーブル駆動テスト
- モック生成・設定・検証、および `testify` 等のアサーション利用
- AST を中心とした静的解析と、ファイル・パッケージ単位の CLI 実行

### 初版の判定方針

- 検出できる構文上の根拠がある場合に限り警告する。
- マーカーが存在する場合に限り、Arrange、Act、Assert の順序が逆転している場合を違反とする。
- マーカーの欠落自体は違反としない。
- 同じフェーズのマーカーは複数回許可し、全体のフェーズ順が崩れていない限り警告しない。
- マーカーは大文字・小文字を区別せず、`// Arrange: input setup` のような補足付き表記も認識する。
- テスト形式ごとに解析不能な場合は、違反ではなく未判定として扱う。

## 8. ユーザーフロー

1. 開発者が対象の Go ファイルまたはパッケージを CLI に指定する。
2. linter がテストコードを解析する。
3. AAA 違反があれば、デフォルトではファイル名・行番号・診断内容を標準エラーに表示する。
4. `--format sarif` が指定された場合は SARIF 形式で出力する。
5. 違反がない場合は終了コード 0、違反がある場合は終了コード 1 で終了する。
6. 開発者が診断内容をもとにテストを修正し、再実行する。

## 9. 機能要件

### CLI

- `goaaa` コマンドとして提供し、ファイル、ディレクトリ、またはパッケージを入力として受け付ける。
- `-h` 等で利用方法を表示できる。
- 診断には少なくとも対象ファイル、行番号、ルール識別子、理由を含める。
- デフォルトは人間向けテキストを標準エラーへ出力する。
- `--format sarif` で SARIF 2.1.0 出力を選択できる。
- 違反の有無を終了コードで表現する。違反なしは 0、違反ありは 1 とする。
- 構文エラーや入力パス不正などの解析エラーは終了コード 2 とする。

### 解析

- Go の構文を正しく解析し、構文エラーを診断できる。
- テスト関数とサブテストのスコープを区別し、それぞれの本体を独立して検査する。
- ヘルパー関数、別パッケージ、テスト関数以外の関数は追跡しない。
- テーブルの各ケースに対するマーカーの位置関係を可能な範囲で確認する。
- アサーションやモックのライブラリ名・API には依存せず、マーカーの位置と順序だけを検査する。
- 同一テスト内に複数の Act / Assert がある場合の扱いを定義し、誤検出を抑制する。
- 解析不能な構造を無理に AAA のいずれかへ分類しない。

### 診断

- 診断メッセージは、違反したフェーズと期待される順序を説明する。
- 同一原因に対して過剰な重複警告を出さない。
- ルール識別子を持ち、将来の設定や抑制に利用できる形式にする。

## 10. 制約と依存関係

- Go の標準ライブラリのみを使用する。
- 主な解析手段は `go/ast`、`go/parser`、必要に応じて `go/token` とする。
- 外部ライブラリの API や型情報に依存しない。
- CLI の対象 Go バージョンを実装前に定義する。

## 11. 成功指標

- 正常な AAA 形式の代表的なテストを警告なしで通過する。
- Arrange・Act・Assert マーカーの明らかな順序違反を検出する。
- テーブル駆動、サブテスト、モック利用の代表例で実行可能な診断を出せる。
- 解析不能な曖昧なケースを、誤った違反として大量に報告しない。
- 診断だけで開発者が修正箇所と問題理由を特定できる。
- 代表的な fixture の範囲で、誤検出・未検出とも 90% 以上を目標とする。

## 12. リスクと例外

- 任意の関数呼び出しを Arrange・Act・Assert に一意に分類することは困難であり、静的解析だけでは判定できないケースがある。
- アサーション用ライブラリやモック実装ごとに API が異なるため、ライブラリ固有の完全な意味理解は保証しない。
- テーブル駆動テストでは、ループ本体とケース定義のどちらを Arrange とみなすかが曖昧になり得る。
- ヘルパー関数に処理を移したテストでは、呼び出し先の意味まで追跡しない場合、誤検出または未検出が生じる。
- 複数の Act / Assert を許容するテストでは、単一の AAA 形式を強制しすぎない。

## 13. リリース計画

### 初版

- CLI の基本入力、終了コード、診断形式を実装する。
- AST ベースの AAA 判定ルールを限定的に実装する。
- 正常系、明らかな違反、テーブル駆動、サブテスト、モック利用の fixture を用意する。
- 代表ケースに対する単体テストと CLI の結合テストを実施する。

### 次期候補

- 設定ファイルによるルールの有効・無効化や許容範囲の指定
- JSON 等の機械可読な出力
- `golangci-lint` などへの統合
- コメント・空行・明示マーカーを利用した判定精度の向上

## 14. 未解決事項

- SARIF 2.1.0 に含める診断ルール・結果の詳細スキーマは実装時に確定する。

## 15. 前提

- AAA は厳密な言語仕様ではなく、静的解析で観測できる構文上のヒューリスティックとして扱う。
- 初版の利用者は、診断が出ないことよりも、明らかな誤警告が少ないことを重視する。
- テストコードの改善を促すことが目的であり、すべてのテストを機械的に単一形式へ変換することは目的としない。

## 16. 付録

### 用語

- Arrange：テスト対象、入力値、依存関係、モックなどを準備する段階
- Act：テスト対象の処理を実行する段階
- Assert：戻り値、エラー、状態、モック呼び出しなどを検証する段階
- AAA：Arrange–Act–Assert の略称

### 代表的な検証例

```go
func TestAdd(t *testing.T) {
	// Arrange
	a, b := 1, 2

	// Act
	got := Add(a, b)

	// Assert
	if got != 3 {
		t.Fatalf("got %d, want 3", got)
	}
}
```
